# Parquet MCP Server - Performance Fixes

**Date**: 2026-01-14  
**Changes**: Applied recommendations to fix massive output file generation

## Changes Applied

### 1. Compact JSON Serialization ✅

**Problem**: Responses were pretty-printed with `indent=2`, causing:
- 16,065 records × ~43 fields = ~690,000 lines
- 28.6 MB file size for what should be ~8-10 MB

**Fix**: Changed all `json.dumps()` calls in `read_parquet` handler to use compact format:
- **Before**: `json.dumps(..., indent=2)`
- **After**: `json.dumps(..., separators=(',', ':'), ensure_ascii=False)`

**Impact**: 
- File size reduction: 28.6 MB → ~8-10 MB (for 16,065 records)
- Line count reduction: 690,801 lines → ~16,065 lines (one per record)
- Faster serialization, lower memory usage

### 2. Pagination Support ✅

**Problem**: No `offset` parameter for pagination, forcing clients to read entire dataset

**Fix**: Added `offset` parameter to tool schema and handler:
- **Tool Schema**: Added `offset` parameter (integer, default: 0, minimum: 0)
- **Handler**: Implemented offset-based pagination using `df.iloc[offset:offset + limit]`
- **Response**: Added `has_more` flag to indicate if more data is available

**Usage**:
```python
# First page
read_parquet(data_type="tasks", limit=10000, offset=0)

# Next page
read_parquet(data_type="tasks", limit=10000, offset=10000)
```

### 3. Maximum Limit Enforcement ✅

**Problem**: No limit on maximum rows, allowing requests that generate huge responses

**Fix**: 
- Added `maximum: 10000` to `limit` parameter in tool schema
- Enforced limit in handler: `limit = min(limit, 10000)`
- Added warning message when limit is capped

**Impact**: Prevents accidental huge requests, encourages pagination

### 4. Improved Response Structure ✅

**Before**:
```json
{
  "count": 16065,
  "total_rows": "16065+",
  "data": [...]
}
```

**After**:
```json
{
  "count": 10000,
  "total_rows": 16065,
  "offset": 0,
  "limit": 10000,
  "has_more": true,
  "data": [...]
}
```

**Benefits**:
- Accurate `total_rows` count (integer, not string)
- Clear pagination information (`offset`, `limit`, `has_more`)
- Better client-side pagination support

  - `total_rows` is now always an integer (not a string like "16065+")
